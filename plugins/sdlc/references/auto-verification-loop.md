# 自動驗證與修復循環

> 適用於需求文件（FRD）和功能規格書（BFS）寫入後的品質保證

## 流程

```
文件寫入磁碟 → 執行規格驗證 → 判斷結果 → 修復（若需要）→ 重新驗證
                                                    ↑                    |
                                                    └────────────────────┘
                                                      （最多 3 次循環）
```

## 判斷標準

| 結果 | 條件 | 處理 |
|------|------|------|
| ✅ 通過 | 0 CRITICAL, 0 WARNING | 輸出完成摘要，建議下一步 |
| ⚠️ 有條件通過 | 0 CRITICAL, >0 WARNING | 自動修復 WARNING → 重新驗證 |
| ❌ 不通過 | >0 CRITICAL | 自動修復 CRITICAL → 重新驗證 |

## 修復策略（依文件類型）

### 需求文件（FRD / CR-FRD）

| 缺失類型 | 修復方式 |
|----------|----------|
| 章節缺失 | 根據範本補充缺失章節 |
| 使用者故事缺 Given-When-Then | 從業務描述推導並補充 |
| 驗收標準不足 | 根據使用者故事生成測試案例 |
| 畫面欄位定義不完整 | 從資料庫結構補充欄位規格 |

### 功能規格書（BFS / CR-BFS）

| 缺失類型 | 修復方式 |
|----------|----------|
| 欄位型態/長度與 DB 不符 | 用 MCP 查 DB，以 DB 為準更新 ER Diagram 和欄位對應表 |
| 規格書遺漏 DB 欄位 | 從 DB 欄位清單補充到欄位對應表，標記 `[DB 補充]` |
| NOT NULL 與必填不一致 | 以 DB 為準，同步更新 Request 必填標記和 VR 規則 |
| VR MaxByteLength 與 varchar 不符 | 以 DB varchar 長度為準更新 VR 規則 |
| API 端點缺 JSON 範例 | 根據 Request/Response 模型生成擬真 JSON |
| 欄位對應表不完整 | 從 ER Diagram 和 DB 結構補充四欄對應 |
| 驗證規則缺失 | 從必填欄位和業務規則推導 VR/BR |
| 舊系統驗證未涵蓋 | 從舊系統分析報告補充缺失的驗證規則 |
| 測試案例不足 | 根據 Handler 和驗證規則生成測試案例 |

## 循環限制

- **最多 3 次**修復循環
- 超過 3 次仍未通過 → 輸出剩餘問題，請使用者決定
