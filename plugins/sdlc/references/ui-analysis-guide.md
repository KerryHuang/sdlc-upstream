# UI 畫面分析指南

> 使用 `chrome-devtools` MCP 工具分析現有系統畫面，提升需求分析與系統分析的正確性。

## 前置條件與偵測

檢查可用工具清單中是否存在 `mcp__chrome-devtools__` 前綴的工具。

- **可用** → 詢問使用者是否提供系統 URL 進行畫面分析
- **不可用** → 跳過畫面分析步驟，依原流程進行

## 適用場景

| 場景 | 說明 | 觸發條件 |
|------|------|---------|
| 功能調整需求 | 分析現有畫面的欄位配置、操作流程 | `--change` 模式，或使用者提到「現有功能」 |
| 系統分析 | 分析現有畫面呼叫的 API、Request/Response 格式 | 分析涉及既有功能的資訊流/資料流 |
| 欄位比對 | 畫面欄位 vs DB 欄位 vs 規格書欄位 | 確認欄位對應正確性 |
| 操作流程驗證 | 實際操作畫面確認業務流程 | FRD 中描述的流程需驗證 |

## 分析流程

### Phase 1：導航與登入

```
1. navigate_page → 目標 URL
2. 若需登入 → take_snapshot 取得表單結構 → fill 帳密 → click 登入
3. 若有環境選擇頁面 → 選擇對應環境
4. 導航到目標功能頁面
```

**注意**：登入憑證由使用者提供，**不儲存、不記錄**到任何文件中。

### Phase 2：畫面結構分析

| 步驟 | 工具 | 產出 |
|------|------|------|
| 截圖 | `take_screenshot` | 視覺化的畫面佈局 |
| DOM 結構 | `take_snapshot` | 精確的欄位名稱、元件類型、層級結構 |
| 表單欄位 | 從 snapshot 提取 | 欄位清單（名稱、類型、必填、預設值） |
| 按鈕/操作 | 從 snapshot 提取 | 可用操作清單 |

**產出格式**：

```markdown
### 畫面分析：{頁面名稱}

**URL**：`{當前 URL}`

#### 欄位清單

| 欄位名稱 | 元件類型 | 必填 | 目前值/預設值 | 備註 |
|----------|----------|------|-------------|------|
| {名稱} | Input/Select/DatePicker | ✅/- | {值} | {備註} |

#### 操作按鈕

| 按鈕 | 位置 | 推測功能 |
|------|------|---------|
| {名稱} | 頂部工具列/表格行內 | {功能} |
```

### Phase 3：API 攔截分析

分析現有功能的實際 API 呼叫，用於驗證資訊流與資料流：

```
1. navigate_page 到目標頁面（觸發初始 API 呼叫）
2. list_network_requests → 取得所有 API 呼叫清單
3. 篩選業務相關 API（排除靜態資源、CDN、auth）
4. get_network_request → 取得關鍵 API 的 Request/Response 詳情
```

**產出格式**：

```markdown
### API 攔截結果：{頁面/操作名稱}

#### 呼叫清單

| # | 方法 | URL | 狀態 | 說明 |
|---|------|-----|------|------|
| 1 | GET | /api/v3/sales/order/paged | 200 | 分頁查詢 |
| 2 | GET | /api/v3/sales/customer/light | 200 | 客戶下拉選單 |

#### 關鍵 API 詳情

**GET /api/v3/sales/order/paged**
- Request: `{ currentPage: 1, pagesSize: 20 }`
- Response 結構摘要:
  - `items[]`: 訂單清單
  - `totalCount`: 總筆數
  - 關鍵欄位: orderId, orderNo, status, customerName, ...
```

### Phase 4：操作流程錄製

模擬使用者操作，記錄每步的畫面變化與 API 呼叫：

```
1. 執行操作（如點擊「新增」）
2. take_screenshot → 記錄畫面變化
3. list_network_requests → 記錄觸發的 API
4. 重複直到操作完成
```

**產出格式**：

```markdown
### 操作流程：{流程名稱}

| 步驟 | 操作 | 畫面變化 | 觸發 API | 備註 |
|------|------|---------|---------|------|
| 1 | 點擊「新增」 | 開啟空白表單 | - | |
| 2 | 選擇客戶 | 下拉選單載入 | GET /customer/light | |
| 3 | 點擊「儲存」 | 返回列表 | POST /order | |
```

## 交叉比對

畫面分析結果應與以下資料源交叉比對：

| 比對來源 | 比對項目 | 不一致處理 |
|---------|---------|-----------|
| DB 結構（MCP 查詢） | 欄位名稱、型別、長度 | 以 DB 為準 |
| 現有程式碼（Entity/Response） | 欄位對應、巢狀結構 | 以程式碼為準 |
| FoxPro 舊系統（若有） | 功能對齊、欄位對照 | 記錄差異 |
| 規格書（若有） | 欄位定義、業務規則 | 標記為規格書缺失 |

## 限制與注意事項

- **不記錄憑證**：登入帳密不寫入任何文件
- **瀏覽器狀態**：MCP 操作的是實際瀏覽器，操作會影響使用者的瀏覽器視窗
- **SPA 路由**：Vue/React SPA 的頁面切換需用 `click` 操作導航，非 `navigate_page`
- **網路延遲**：頁面載入後稍等再截圖，確保資料已載入
- **分析結果用途**：畫面分析結果作為需求/系統分析的輸入證據，不直接寫入最終文件的截圖
